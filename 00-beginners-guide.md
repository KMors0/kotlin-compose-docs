## Фундамент: как вообще работает код

### Что такое программа?

Программа — это набор инструкций, которые компьютер выполняет по порядку. Представь кулинарный рецепт: «взять яйцо → разбить → вылить на сковороду → жарить 3 минуты». Программа работает так же, только вместо яиц — данные, а вместо сковороды — процессор.

Код (исходный код) — это текст, написанный на языке программирования. Компьютер не понимает его напрямую, поэтому существует **компилятор** — программа-переводчик, которая превращает твой код в машинные инструкции (нули и единицы), которые процессор уже может выполнить.

### Что такое язык программирования?

Язык программирования — это способ говорить с компьютером. Их сотни, но принцип один: ты пишешь инструкции понятным человеку текстом, а компилятор переводит их в машинный код.

**Kotlin** — один из таких языков. Он компилируется (переводится) в байт-код JVM (виртуальной машины Java), который работает на Android, серверах и десктопе. А ещё Kotlin умеет компилироваться в нативный код для iOS и в JavaScript/WebAssembly для браузера.

### Что такое JVM?

JVM (Java Virtual Machine) — это виртуальная машина, программа, которая выполняет скомпилированный Java/Kotlin-код. Аналогия: JVM — это как DVD-плеер. Ты можешь записать диск (скомпилировать код) один раз, а потом воспроизвести его на любом плеере (на любом устройстве с JVM). Поэтому Kotlin-программы работают везде, где есть JVM.

### Что такое API?

API (Application Programming Interface) — это «меню» программы или сервиса. Когда ты приходишь в ресторан, ты не идёшь на кухню — ты выбираешь из меню, и официант передаёт заказ. API работает так же: программа (или сервер) предлагает набор действий, которые ты можешь вызвать, не зная, как они устроены внутри.

Например, API погодного сервиса может предоставить функцию `getWeather(city)`. Ты вызываешь её — получаешь температуру. Как именно сервер её узнаёт — тебя не касается.

### Что такое библиотека и фреймворк?

**Библиотека** — это набор готовых инструментов, который ты используешь, когда нужно. Аналогия: библиотека — как набор инструментов в гараже. Ты берёшь молоток, когда нужно забить гвоздь, и кладёшь обратно.

**Фреймворк** — это каркас, в котором ты работаешь. Аналогия: фреймворк — как конструктор LEGO с базовой платформой. Платформа уже задаёт общую структуру, а ты добавляешь детали. Фреймворк сам вызывает твой код в нужные моменты, а не ты вызываешь функции фреймворка.

- **Kotlin stdlib** — библиотека (берёшь нужные функции)
- **Compose Multiplatform** — фреймворк (он управляет вызовами твоего UI-кода)

## Переменные и типы данных

### Что такое переменная?

Переменная — это именованная коробка, в которой хранится значение. Аналогия: переменная — как ящик с этикеткой. На этикетке написано «возраст», внутри лежит число 25. Ты можешь заглянуть в ящик (прочитать значение) или положить другое число (изменить значение).

В Kotlin переменные объявляются так:

```kotlin
val name = "Алиса"    // val — значение нельзя менять (как запечатанный конверт)
var age = 25           // var — значение можно менять (как ящик с этикеткой)

age = 26               // ОК, var можно перезаписать
// name = "Борис"      // ОШИБКА! val перезаписать нельзя
```

**Правило:** по умолчанию используй `val`. Если точно знаешь, что значение будет меняться — используй `var`. Это делает код предсказуемее.

### Типы данных

Тип данных определяет, **что** лежит в переменной — число, текст, правда/ложь и т.д. Kotlin — статически типизированный язык: тип каждой переменной определяется один раз и не меняется.

| Тип | Что хранит | Пример |
|-----|-----------|--------|
| `Int` | Целое число | `42`, `-7`, `0` |
| `Long` | Очень большое целое число | `9999999999L` |
| `Float` | Дробное число (меньше точности) | `3.14f` |
| `Double` | Дробное число (больше точности) | `3.14159265` |
| `String` | Текст (строка) | `"Привет, мир!"` |
| `Boolean` | Правда или ложь | `true`, `false` |
| `Char` | Один символ | `'А'`, `'7'` |
| `Unit` | Ничего (аналог void) | Возвращается функцией, которая ничего не возвращает |

Kotlin умеет автоматически определять тип (это называется **вывод типа**):

```kotlin
val x = 42          // Kotlin понимает: это Int
val name = "Мария"  // Kotlin понимает: это String
```

Но можно указать тип явно, если нужно:

```kotlin
val x: Int = 42
val name: String = "Мария"
```

## Функции

### Что такое функция?

Функция — это именованный блок инструкций, который можно вызывать сколько угодно раз. Аналогия: функция — как рецепт блюда. Вместо того чтобы каждый раз расписывать шаги, ты говоришь «приготовь борщ» — и повар знает, что делать.

Функция может **принимать аргументы** (входные данные) и **возвращать результат**:

```kotlin
// Объявление функции
fun greet(name: String): String {
    return "Привет, $name!"
}

// Вызов функции
val message = greet("Алиса")  // message = "Привет, Алиса!"
```

Разберём по частям:
- `fun` — ключевое слово, означает «это функция»
- `greet` — имя функции
- `(name: String)` — аргумент: функция ждёт текст (String)
- `: String` — тип возвращаемого значения (функция отдаст текст)
- `return` — что именно вернуть

### Функция-выражение (однострочная)

Если функция простая, её можно записать короче:

```kotlin
fun double(x: Int): Int = x * 2
```

Это то же самое, что:

```kotlin
fun double(x: Int): Int {
    return x * 2
}
```

### Функция без возвращаемого значения

Если функция ничего не возвращает, тип результата — `Unit` (его можно не писать):

```kotlin
fun sayHello() {
    println("Привет!")
}
```

### Аргументы по умолчанию

Можно задать значение по умолчанию — тогда при вызове можно не указывать этот аргумент:

```kotlin
fun greet(name: String, greeting: String = "Привет"): String {
    return "$greeting, $name!"
}

greet("Алиса")               // "Привет, Алиса!"
greet("Алиса", "Здравствуй")  // "Здравствуй, Алиса!"
```

## Классы и объекты

### Что такое класс?

Класс — это **чертёж** (шаблон, описание). Он определяет, какие данные и действия будут у объектов этого типа. Аналогия: класс — это чертёж автомобиля на бумаге. На чертеже указано: «у машины есть цвет, марка и скорость; машина может поехать и затормозить». Но сам чертёж — это не машина. Это описание того, как сделать машину.

### Что такое объект (инстанс)?

Объект — это конкретная реализация класса. Аналогия: если класс — чертёж автомобиля, то объект — конкретная машина, стоящая перед тобой. У неё есть конкретный цвет (красный), конкретная марка (Toyota), конкретная скорость (0 км/ч).

```kotlin
// Класс — чертёж
class Car(val brand: String, val color: String) {
    var speed: Int = 0

    fun drive() {
        speed = 60
        println("$brand едет со скоростью $speed км/ч")
    }

    fun stop() {
        speed = 0
        println("$brand остановилась")
    }
}

// Объект — конкретная машина
val myCar = Car("Toyota", "Красный")    // создали объект (инстанс)
val friendsCar = Car("BMW", "Синий")     // ещё один объект

myCar.drive()     // "Toyota едет со скоростью 60 км/ч"
friendsCar.stop()  // "BMW остановилась"
```

### Конструктор

**Конструктор** — это инструкция, которая вызывается при создании объекта. Он инициализирует объект — задаёт начальные значения. Аналогия: конструктор — как приёмщик на заводе, который записывает параметры заказа «цвет = красный, марка = Toyota» перед тем, как машина сойдёт с конвейера.

В Kotlin первичный конструктор объявляется прямо в заголовке класса:

```kotlin
class Car(val brand: String, val color: String)
//          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//          Это и есть конструктор: он требует brand и color при создании
```

Также бывают вторичные конструкторы с другой логикой инициализации, но они используются реже.

### Свойства и методы

- **Свойство (property)** — это характеристика объекта (данные). В примере выше: `brand`, `color`, `speed` — свойства.
- **Метод (method)** — это действие, которое объект может выполнить. В примере: `drive()` и `stop()` — методы.

Аналогия: у собаки есть свойства (кличка, порода, возраст) и методы (лаять, бегать, есть).

### Что такое `this`?

`this` — это ссылка на текущий объект. Используется, когда нужно отличить свойство объекта от аргумента функции с таким же именем:

```kotlin
class Person(val name: String) {
    fun introduce() {
        println("Меня зовут ${this.name}")  // this.name — свойство объекта
    }
}
```

Чаще всего `this` можно опустить — Kotlin поймёт сам. Но иногда он нужен для устранения неоднозначности.

## Наследование и интерфейсы

### Что такое наследование?

Наследование — это механизм, который позволяет одному классу перенять свойства и методы другого. Аналогия: «электромобиль» — это «автомобиль», но с дополнительными особенностями (батарея вместо бака). Электромобиль наследует всё, что есть у автомобиля (колёса, руль, тормоза), и добавляет своё.

```kotlin
// Родительский класс (открытый для наследования — нужен open)
open class Animal(val name: String) {
    fun eat() { println("$name ест") }
}

// Дочерний класс наследует Animal
class Dog(name: String) : Animal(name) {
    fun bark() { println("$name лает: Гав!") }
}

val dog = Dog("Рекс")
dog.eat()   // "Рекс ест" — унаследованный метод
dog.bark()  // "Рекс лает: Гав!" — собственный метод
```

По умолчанию классы в Kotlin **нельзя** наследовать — для этого нужно добавить `open`. Это защита от случайного наследования, которое может сломать логику.

### Что такое интерфейс?

Интерфейс — это **контракт**: «если класс подписался, он обязан реализовать эти методы». Аналогия: интерфейс — как вакансия. Там написано «должен уметь программировать и говорить по-английски». Любой кандидат (класс), который откликнулся, должен подтвердить эти навыки.

```kotlin
interface Swimmable {
    fun swim()  // Класс-наследник ОБЯЗАН реализовать этот метод
}

class Duck(name: String) : Animal(name), Swimmable {
    override fun swim() {
        println("$name плывёт")
    }
}

val duck = Duck("Дональд")
duck.eat()   // Унаследовано от Animal
duck.swim()  // Реализовано по контракту Swimmable
```

**В чём разница между наследованием и интерфейсом?**

| | Наследование | Интерфейс |
|---|---|---|
| Суть | «Является» (Dog — это Animal) | «Умеет» (Duck умеет плавать) |
| Сколько? | Только один родитель | Сколько угодно интерфейсов |
| Содержит | И реализацию, и данные | Только объявления (и иногда реализацию по умолчанию) |
| Аналогия | Родство по крови | Профессия/навык |

### Что такое override?

`override` — значит «переопределить». Дочерний класс может изменить поведение метода, который он унаследовал:

```kotlin
open class Animal(val name: String) {
    open fun sound() { println("$name издаёт звук") }
}

class Cat(name: String) : Animal(name) {
    override fun sound() { println("$name мяукает") }
}
```

## Коллекции: List, Set, Map

### Что такое коллекция?

Коллекция — это контейнер, который хранит **несколько значений** в одной переменной. Аналогия: если переменная — это один ящик, то коллекция — это шкаф с множеством ящиков.

В Kotlin три основных вида коллекций: **List**, **Set** и **Map**. Каждый подходит для своей задачи.

### List (Список)

**Что это:** Упорядоченная коллекция элементов, где могут быть дубликаты. Аналогия: список покупок. В нём важен порядок (первый — хлеб, второй — молоко), и можно написать «хлеб» дважды.

```kotlin
// Неизменяемый список (нельзя добавлять/удалять)
val fruits = listOf("Яблоко", "Банан", "Апельсин")
println(fruits[0])        // "Яблоко" — доступ по индексу (номеру)
println(fruits.size)      // 3 — количество элементов

// Изменяемый список
val todo = mutableListOf("Помыть посуду", "Вынести мусор")
todo.add("Покормить кота")       // Добавить в конец
todo.removeAt(0)                  // Удалить первый
todo[0] = "Вынести стекло"        // Заменить по индексу
```

**Когда использовать:** Когда важен порядок элементов и нужен доступ по номеру (индексу).

### Set (Множество)

**Что это:** Коллекция **уникальных** элементов, порядок не гарантирован. Аналогия: множество — как группа людей в комнате. Каждый человек там только один (нет дубликатов), но «порядок» не определён — люди стоят как стоят.

```kotlin
val colors = setOf("Красный", "Синий", "Зелёный", "Красный")
println(colors)  // [Красный, Синий, Зелёный] — дубликат "Красный" удалён

val mutableColors = mutableSetOf("Красный")
mutableColors.add("Синий")
mutableColors.add("Красный")  // Не добавится — уже есть
```

**Когда использовать:** Когда нужно проверить «есть ли элемент?» или гарантировать уникальность (например, список ID пользователей).

### Map (Словарь / Отображение)

**Что это:** Коллекция пар «ключ — значение». Аналогия: словарь. Ты ищешь слово (ключ) — находишь определение (значение). Или телефонная книга: имя (ключ) → номер телефона (значение).

```kotlin
// Неизменяемый Map
val capitals = mapOf(
    "Россия" to "Москва",
    "Франция" to "Париж",
    "Япония" to "Токио"
)
println(capitals["Россия"])   // "Москва" — поиск по ключу

// Изменяемый Map
val scores = mutableMapOf("Алиса" to 95)
scores["Борис"] = 87       // Добавить новую пару
scores["Алиса"] = 98       // Обновить значение
scores.remove("Борис")     // Удалить по ключу
```

**Когда использовать:** Когда нужно связать одни данные с другими (имя → контакт, ID → объект, настройка → значение).

### Сравнение коллекций

| Свойство | List | Set | Map |
|----------|------|-----|-----|
| Порядок | Да (по индексу) | Нет гарантии | Нет гарантии |
| Дубликаты | Разрешены | Запрещены | Ключи уникальны, значения могут повторяться |
| Доступ | По индексу `list[0]` | Через проверку `set.contains()` | По ключу `map["key"]` |
| Аналогия | Список покупок | Группа людей | Телефонная книга / словарь |
| Типичный вопрос | «Какой 3-й элемент?» | «Есть ли этот элемент?» | «Какое значение у этого ключа?» |

### Изменяемые и неизменяемые коллекции

В Kotlin коллекции делятся на **изменяемые** (mutable) и **неизменяемые** (read-only):

- `listOf()` — нельзя менять после создания
- `mutableListOf()` — можно добавлять, удалять, заменять элементы

Это не баг, а фича: по умолчанию коллекция неизменяемая — так безопаснее. Если точно нужно менять — используешь mutable-версию. Это часть общей философии Kotlin: по умолчанию — безопасно.

### Полезные операции с коллекциями

```kotlin
val numbers = listOf(1, 2, 3, 4, 5, 6)

// filter — отобрать элементы по условию
val evens = numbers.filter { it % 2 == 0 }  // [2, 4, 6]

// map — преобразовать каждый элемент
val doubled = numbers.map { it * 2 }         // [2, 4, 6, 8, 10, 12]

// find — найти первый подходящий
val firstBig = numbers.find { it > 4 }       // 5

// sortedBy — отсортировать
val sorted = numbers.sortedByDescending { it }  // [6, 5, 4, 3, 2, 1]

// groupBy — сгруппировать
val grouped = numbers.groupBy { it % 2 }     // {1=[1, 3, 5], 0=[2, 4, 6]}

// forEach — выполнить действие для каждого
numbers.forEach { println(it) }

// reduce — свернуть в одно значение
val sum = numbers.reduce { acc, n -> acc + n }  // 21
```

> [!INFO]
> **`it`** — автоматическое имя единственного параметра лямбды в Kotlin. В выражении `numbers.filter { it % 2 == 0 }` лямбда получает каждый элемент списка, и Kotlin автоматически называет его `it`. Можно дать своё имя: `numbers.filter { number -> number % 2 == 0 }` — но `it` короче и привычнее для коротких лямбд.
>
> **`acc` и `n`** в `reduce { acc, n -> acc + n }` — здесь у лямбды два параметра, поэтому `it` не работает. `acc` — аккумулятор (накопленный результат), `n` — текущий элемент. Имена можно дать любые: `reduce { sum, element -> sum + element }`.
>
> **Цепочка операций:** коллекционные функции можно соединять: `numbers.filter { it > 2 }.map { it * 10 }.forEach { println(it) }` — отфильтровать, потом каждый умножить, потом каждый напечатать. Каждый `it` относится к своему этапу цепочки.

## Null Safety — безопасность от пустоты

### Что такое null?

`null` — это «ничего», «пусто», «значение отсутствует». Аналогия: переменная — ящик с этикеткой. В обычном ящике что-то лежит (число, текст). А `null` означает, что ящик пуст — этикетка есть, а внутри ничего нет.

Проблема `null` в том, что если попытаться вызвать метод у `null`, программа упадёт с ошибкой `NullPointerException` (NPE). Это самая частая ошибка в Java и многих других языках. В Kotlin система **null safety** защищает от этого на уровне компилятора.

### Nullable и non-nullable типы

В Kotlin типы разделены на две группы:

```kotlin
val name: String = "Алиса"     // Non-nullable: НЕ может быть null
// name = null                  // ОШИБКА КОМПИЛЯЦИИ!

val nickname: String? = null   // Nullable: МОЖЕТ быть null (знак ?)
nickname = "Аля"               // Тоже ОК
```

Знак `?` после типа означает: «здесь может быть null». Без `?` — компилятор гарантирует, что null там не будет.

### Операторы для работы с null

| Оператор | Название | Что делает | Пример |
|----------|----------|-----------|--------|
| `?.` | Безопасный вызов | Вызывает метод, если не null; иначе возвращает null | `name?.length` |
| `?:` | Оператор Элвиса | Возвращает левое, если не null; иначе правое | `name ?: "Безымянный"` |
| `!!` | Утверждение не-null | «Я гарантирую, что тут не null!» — если null, упадёт | `name!!.length` |
| `as?` | Безопасное приведение | Приводит тип или возвращает null | `obj as? String` |

**Примеры:**

```kotlin
val name: String? = null

// Безопасный вызов — не упадёт
val length = name?.length     // length = null

// Оператор Элвиса — значение по умолчанию
val display = name ?: "Аноним"  // display = "Аноним"

// Комбинация
val safeLength = name?.length ?: 0  // safeLength = 0

// Опасный вызов — упадёт при null!
// val crash = name!!.length       // NullPointerException!
```

**Правило:** Используй `?.` и `?:` по умолчанию. `!!` — только когда ты на 100% уверен, что null там не будет (и даже тогда — подумай дважды).

## Асинхронное программирование

### Что такое синхронный код?

Синхронный код выполняется **строго по порядку**, строка за строкой. Каждая строка ждёт, пока предыдущая завершится:

```kotlin
fun main() {
    val user = fetchUser()        // Шаг 1: ждём 2 секунды (запрос к серверу)
    val posts = fetchPosts()      // Шаг 2: ждём ещё 2 секунды
    println("Готово!")            // Шаг 3: всего через 4 секунды
}
```

Аналогия: стоишь в очереди в магазине. Первый покупатель расплачивается — ты ждёшь. Потом второй — ты ждёшь. Только когда дойдёт до тебя — ты действуешь.

Проблема: если `fetchUser()` занимает 2 секунды (запрос по сети), то программа **ничего не делает** всё это время — просто ждёт. А мог бы делать полезную работу.

### Что такое асинхронный код?

Асинхронный код позволяет **не ждать** завершения долгой операции, а продолжать делать что-то параллельно. Когда операция завершится — результат обработается.

```kotlin
// Асинхронно: обе загрузки идут параллельно
suspend fun loadDashboard() = coroutineScope {
    val deferredUser = async { fetchUser() }    // Запустили
    val deferredPosts = async { fetchPosts() }   // Запустили параллельно
    val user = deferredUser.await()              // Дождались первое
    val posts = deferredPosts.await()            // Дождались второе
    println("Готово!")                           // Всего ~2 секунды вместо 4!
}
```

Аналогия: ты заказал пиццу и суши одновременно. Пока готовят, ты смотришь сериал. Когда привезут — поешь. Не нужно ждать пиццу, чтобы заказать суши.

### Зачем это нужно?

В реальных приложениях почти всё — долгие операции:
- Загрузка данных из интернета (сотни миллисекунд — секунды)
- Чтение/запись файлов (миллисекунды — секунды)
- Запросы к базе данных (миллисекунды)
- Ожидание ответа от GPS-датчика

Если бы приложение работало синхронно, интерфейс «замораживался» бы при каждом запросе к серверу. Асинхронность позволяет UI оставаться отзывчивым.

### Поток (Thread) vs Корутина (Coroutine)

**Поток (Thread)** — это независимая линия выполнения, управляемая операционной системой. Каждый поток — как отдельный рабочий на заводе. Потоки дорогие: создание одного потока занимает ~1 МБ памяти, переключение между потоками — дорогостоящая операция.

**Корутина (Coroutine)** — это лёгкая «подзадача» внутри потока. Корутины не привязаны к конкретному потоку: они могут приостанавливаться и возобновляться, позволяя одному потоку выполнять много корутин. Аналогия: один рабочий (поток) может готовить суп (корутина 1), пока варится — нарезать салат (корутина 2), пока салат настаивается — проверить суп.

| | Поток (Thread) | Корутина (Coroutine) |
|---|---|---|
| Создание | Дорого (~1 МБ) | Дёшево (~сотни байт) |
| Количество | Десятки-сотни | Сотни тысяч |
| Переключение | Операционная система | Сама программа (быстрее) |
| Аналогия | Отдельный рабочий | Список задач одного рабочего |

## Корутины подробнее

### Что такое корутина?

Корутина — это блок кода, который можно **приостановить** (suspend) и **возобновить** (resume) позже, не блокируя поток. Когда корутина приостанавливается (например, ждёт ответа от сервера), поток освобождается и может выполнять другую корутину.

### Как запустить корутину?

```kotlin
// Способ 1: launch — «запусти и забудь»
viewModelScope.launch {
    val data = fetchData()    // Приостанавливается, не блокирует
    showData(data)            // Возобновляется, когда данные готовы
}

// Способ 2: async — «запусти и верни результат»
val deferred = viewModelScope.async {
    fetchData()
}
val result = deferred.await()  // Дождаться результата
```

- **launch** — для действий, где результат не нужен (показать уведомление, записать лог)
- **async** — для действий, где нужен результат (получить данные с сервера)

### Что такое suspend-функция?

`suspend` — ключевое слово, которое говорит: «эту функцию можно приостанавливать». Такие функции можно вызывать **только из корутины или другой suspend-функции**:

```kotlin
suspend fun fetchUser(): User {
    delay(1000)          // Приостановка на 1 секунду (не блокирует поток!)
    return User("Алиса")
}

// ОШИБКА: нельзя вызвать suspend-функцию из обычного кода
// fun main() {
//     fetchUser()
// }

// ПРАВИЛЬНО: из корутины
fun main() = runBlocking {
    val user = fetchUser()  // ОК
}
```

### Что такое Dispatcher?

Dispatcher определяет, **на каком потоке** будет работать корутина:

| Dispatcher | Для чего | Аналогия |
|------------|----------|----------|
| `Dispatchers.Main` | Работа с UI (обновление экрана) | Прилавок — общение с покупателем |
| `Dispatchers.IO` | Сетевые запросы, работа с файлами, БД | Склад — тяжёлая работа в фоне |
| `Dispatchers.Default` | Вычисления (сортировка, математика) | Офис — интеллектуальная работа |

```kotlin
viewModelScope.launch(Dispatchers.IO) {
    val data = fetchData()         // На IO-потоке (сетевой запрос)
    withContext(Dispatchers.Main) {
        textView.text = data       // Переключились на Main (обновляем UI)
    }
}
```

### Что такое Structured Concurrency?

Это правило: **корутины живут в иерархии**. У каждой корутины есть «родитель». Если родитель отменяется — все его «дети» тоже отменяются. Аналогия: если закрывается ресторан, все заказы с кухни автоматически отменяются.

```kotlin
viewModelScope.launch {           // Родитель
    launch {                      // Ребёнок 1
        delay(5000)
        println("Это может не выполниться")
    }
    launch {                      // Ребёнок 2
        delay(3000)
        println("И это тоже")
    }
}
// Когда viewModelScope отменяется (пользователь ушёл с экрана),
// обе дочерние корутины тоже отменяются автоматически
```

Это предотвращает утечки: если пользователь закрыл экран, корутины не продолжат работать в фоне.

## Лямбды и функции высшего порядка

### Что такое лямбда?

Лямбда — это **безымянная функция**, которую можно передать как значение. Аналогия: обычная функция — как рецепт с названием «Борщ» в кулинарной книге. Лямбда — как рецепт, написанный на стикере: имени нет, но приготовить можно.

```kotlin
// Обычная функция
fun double(x: Int): Int = x * 2

// Та же логика как лямбда
val double: (Int) -> Int = { x -> x * 2 }

// Короткая запись (it — единственный аргумент)
val double: (Int) -> Int = { it * 2 }

// Вызов
double(5)  // 10
```

### Зачем нужны лямбды?

Они позволяют **передавать поведение как данные**. Ты не заранее решаешь «что делать», а даешь программе инструкцию, которую она выполнит когда нужно:

```kotlin
val numbers = listOf(1, 2, 3, 4, 5)

// Лямбда { it % 2 == 0 } — это условие фильтрации
val evens = numbers.filter { it % 2 == 0 }  // [2, 4]

// Лямбда { it * it } — это правило преобразования
val squares = numbers.map { it * it }        // [1, 4, 9, 16, 25]
```

### Что такое функция высшего порядка?

Это функция, которая **принимает другую функцию как аргумент** или **возвращает функцию**. `filter`, `map`, `forEach` — всё это функции высшего порядка, потому они принимают лямбду.

```kotlin
// Своя функция высшего порядка
fun doTwice(action: () -> Unit) {
    action()
    action()
}

doTwice { println("Привет!") }
// Выведет:
// Привет!
// Привет!
```

## Обобщения (Generics)

### Что такое Generics?

Generics (обобщения) — это способ написать код, который работает с **разными типами**, не дублируя логику. Аналогия: коробка с этикеткой «Коробка<T>». Вместо того чтобы делать отдельные классы «КоробкаДляЯблок», «КоробкаДляАпельсинов», ты делаешь одну универсальную коробку и пишешь, что внутри:

```kotlin
// Без Generics — нужна отдельная коробка для каждого типа
class StringBox(val value: String)
class IntBox(val value: Int)

// С Generics — одна коробка для всех типов
class Box<T>(val value: T)

val stringBox = Box("Привет")      // T = String
val intBox = Box(42)                // T = Int
val userList = Box(listOf("Алиса")) // T = List<String>
```

`T` — это «переменная типа». Она подставляется при использовании. Стандартные коллекции используют Generics:

```kotlin
val names: List<String> = listOf("Алиса", "Борис")  // List, где T = String
val scores: Map<String, Int> = mapOf("Алиса" to 95)  // Map, где K=String, V=Int
```

Без Generics компилятор не знал бы, что лежит в коллекции, и тебе пришлось бы вручную приводить типы (и получать ошибки в runtime). С Generics ошибка будет найдена при компиляции.

## Sealed-классы и Data-классы

### Data-класс

`data class` — это класс, специально созданный для **хранения данных**. Он автоматически генерирует полезные методы: `equals()` (сравнение), `hashCode()` (для хэш-таблиц), `toString()` (текстовое представление), `copy()` (создание копии с изменениями), `componentN()` (деструктуризация).

Аналогия: data class — как запись в таблице. Строка «Имя = Алиса, Возраст = 25» — это данные, и всё, что нужно — уметь их сравнивать, копировать и красиво выводить.

```kotlin
data class User(val name: String, val age: Int)

val alice = User("Алиса", 25)
val alice2 = User("Алиса", 25)

println(alice == alice2)           // true (автоматическое сравнение по содержимому)
println(alice)                      // User(name=Алиса, age=25) (автоматический toString)
val olderAlice = alice.copy(age=26) // Копия с изменением
```

### Sealed-класс

`sealed class` — это класс, у которого **ограниченное число подклассов**. Все подклассы должны быть объявлены в том же файле. Аналогия: светофор имеет строго три состояния: красный, жёлтый, зелёный. Никаких «серо-буро-малиновых» — только эти три. sealed class гарантирует, что ты обработаешь ВСЕ возможные варианты.

```kotlin
sealed class Result {
    data class Success(val data: String) : Result()
    data class Error(val message: String) : Result()
    object Loading : Result()
}

fun handleResult(result: Result) = when (result) {
    is Result.Success -> println("Данные: ${result.data}")
    is Result.Error   -> println("Ошибка: ${result.message}")
    Result.Loading    -> println("Загрузка...")
    // Если забудешь один из вариантов — компилятор выдаст ошибку!
}
```

Это мощный инструмент для моделирования состояний в приложении (загрузка → успех / ошибка) и в архитектурных паттернах (MVI).

## Scope-функции: let, apply, run, also, with

### Что такое scope-функции?

Scope-функции — это функции, которые создают **временную область видимости** для объекта. Внутри этой области можно обращаться к объекту без повторения его имени. Аналогия: представь, что ты работаешь с документом. Вместо того чтобы каждый раз говорить «в документе номер 1 поставь дату, в документе номер 1 добавь подпись», ты говоришь: «Возьми документ 1. В нём: поставь дату, добавь подпись».

### let — «дай мне объект, я что-то с ним сделаю»

Главное: обращение через `it`, возвращает результат лямбды.

```kotlin
val name: String? = getName()
name?.let {
    // Этот блок выполнится только если name не null
    println("Имя: $it, длина: ${it.length}")
}
```

Самое частое применение — проверка на null. `?.let { }` — «если не null, сделай что-то».

### apply — «настрой объект»

Главное: обращение через `this` (можно опустить), возвращает сам объект.

```kotlin
val person = Person().apply {
    name = "Алиса"    // Вместо this.name
    age = 25          // Вместо this.age
}
// apply вернёт тот же объект person, но уже с заполненными полями
```

Идеально для настройки объекта при создании.

### also — «сделай что-то дополнительно»

Главное: обращение через `it`, возвращает сам объект. Похоже на `let`, но возвращает объект, а не результат лямбды.

```kotlin
val numbers = mutableListOf(1, 2, 3).also {
    println("Исходный список: $it")  // Побочный эффект (логирование)
}
// numbers = [1, 2, 3] — also вернул сам список
```

Часто используется для логирования и отладки.

### run — «выполни блок и верни результат»

Главное: обращение через `this`, возвращает результат лямбды.

```kotlin
val greeting = "Привет".run {
    length                    // this.length (this опущен)
    uppercase()               // this.uppercase()
}
```

### with — «с объектом сделай следующее»

Похоже на `run`, но объект передаётся как аргумент, а не как получатель:

```kotlin
val result = with(person) {
    "$name, $age лет"
}
```

### Шпаргалка по scope-функциям

| Функция | Обращение | Возвращает | Типичное применение |
|---------|-----------|-----------|-------------------|
| `let` | `it` | Результат лямбды | Проверка на null, преобразование |
| `apply` | `this` | Сам объект | Настройка объекта |
| `also` | `it` | Сам объект | Логирование, побочные эффекты |
| `run` | `this` | Результат лямбды | Вычисление на объекте |
| `with` | `this` | Результат лямбды | Группировка операций |

## Архитектурные понятия

### Что такое архитектура приложения?

Архитектура — это способ организации кода. Аналогия: при строительстве дома кто-то решает, где будут стены, где двери, где электричество. Архитектура приложения — это «план дома»: какие части кода за что отвечают и как они связаны.

Без архитектуры код превращается в «спагетти» — всё переплетено, и изменение в одном месте ломает другое.

### Clean Architecture (Чистая архитектура)

Идея: раздели код на **слои**, где каждый слой делает свою работу и не лезет в чужую.

```
┌─────────────────────────────────┐
│  Presentation (UI, ViewModel)   │  ← Пользователь видит это
├─────────────────────────────────┤
│  Domain (бизнес-логика)         │  ← Правила приложения
├─────────────────────────────────┤
│  Data (сеть, база данных)       │  ← Откуда берутся данные
└─────────────────────────────────┘
```

**Правило зависимостей:** Domain не знает о Data и Presentation. Data знает о Domain. Presentation знает о Domain. Это как в ресторане: зал (Presentation) общается с кухней (Domain), а кухня — со складом (Data). Но склад не знает, что происходит в зале.

### MVI (Model-View-Intent)

MVI — это архитектурный паттерн, особенно популярный с Compose. Поток данных однонаправленный:

```
Пользователь нажал кнопку
         ↓
   Intent (намерение)
         ↓
   Reducer (обработчик) → новое State (состояние)
         ↓
   View (экран) отображает State
```

Аналогия: калькулятор. Ты нажимаешь кнопку «+» (Intent), калькулятор вычисляет (Reducer), на экране появляется результат (State → View). Экран никогда не меняет состояние напрямую — только через Intent.

### Dependency Injection (DI)

**DI (внедрение зависимостей)** — это способ организации кода, при котором объекты не создают свои зависимости сами, а получают их «снаружи».

Аналогия: вместо того чтобы каждому работнику ходить на склад за инструментами, инструментами их обеспечивает менеджер. Работнику не важно, откуда инструмент — главное, чтобы он был нужного типа.

```kotlin
// Без DI: класс сам создаёт зависимость
class ViewModel {
    val repository = UserRepository()  // Жёсткая связь — сложно тестировать
}

// С DI: зависимость передаётся извне
class ViewModel(val repository: UserRepository) {
    // Теперь можно подставить мок-репозиторий для тестирования
}
```

**Koin** — популярный DI-фреймворк для Kotlin Multiplatform. Он работает как «реестр»: ты регистрируешь, какие объекты нужны, и Koin автоматически предоставляет их там, где нужно.

## Compose и UI-термины

### Что такое UI?

UI (User Interface, пользовательский интерфейс) — это то, что пользователь видит на экране: кнопки, текст, списки, поля ввода. В программировании UI — это код, который описывает, как выглядит экран.

### Декларативный UI (подход Compose)

Существует два подхода к созданию UI:

**Императивный** (старый подход): ты говоришь **как** изменить экран шаг за шагом. Аналогия: «Возьми синюю краску. Покрась кнопку. Напиши на ней "OK". Теперь измени текст на "Готово"».

```kotlin
// Императивный (не Compose)
button.setText("Готово")
button.setTextColor(Color.BLUE)
button.setEnabled(true)
```

**Декларативный** (подход Compose): ты описываешь, **что** должно быть на экране, а фреймворк сам решает, как это нарисовать. Аналогия: вместо инструкций маляру ты показываешь картинку с готовым результатом.

```kotlin
// Декларативный (Compose)
@Composable
fun MyButton(text: String) {
    Button(
        onClick = { /* действие */ },
        colors = ButtonDefaults.buttonColors(Color.Blue)
    ) {
        Text(text)
    }
}
```

Если `text` меняется, ты просто вызываешь `MyButton("Готово")` — и Compose сам обновит экран.

### Что такое @Composable?

`@Composable` — это аннотация, которая превращает обычную функцию в функцию, рисующую UI. Такие функции можно вкладывать друг в друга, как матрёшки:

```kotlin
@Composable
fun Screen() {                  // Экран
    Column {                    // Вертикальная колонка
        Header()                // Заголовок
        Content()               // Содержимое
        Footer()                // Подвал
    }
}

@Composable
fun Header() {
    Text("Моё приложение")
}
```

### Что такое состояние (State)?

Состояние — это данные, которые **меняются со временем** и влияют на то, что видит пользователь. Аналогия: счётчик на сайте. Когда ты нажимаешь «+», состояние меняется с 0 на 1, и экран обновляется.

```kotlin
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    //        ↑ remember запоминает значение при перерисовке
    //                    ↑ mutableStateOf — изменяемое состояние

    Column {
        Text("Счётчик: $count")
        Button(onClick = { count++ }) {
            Text("+1")
        }
    }
}
```

Когда `count` меняется, Compose автоматически перерисовывает только ту часть UI, которая зависит от `count`.

### Что такое рекомпозиция?

**Рекомпозиция** — это перерисовка (повторный вызов) Composable-функции, когда её состояние изменилось. Это ключевой механизм Compose: ты меняешь данные → Compose автоматически обновляет экран.

Важно: рекомпозиция — это не «полная перерисовка всего экрана». Compose умный: он перерисовывает только те части, которые действительно изменились. Но если писать код неаккуратно, можно заставить Compose перерисовывать больше, чем нужно — это называется «избыточная рекомпозиция» и приводит к подтормаживаниям.

### Что такое expect/actual?

Это механизм Kotlin Multiplatform для написания **платформенно-зависимого кода**:

- `expect` — в общем модуле говорит: «Мне нужна функция с такой сигнатурой, но я не знаю, как она работает на конкретной платформе»
- `actual` — в платформенном модуле говорит: «Вот реализация для этой платформы»

```kotlin
// commonMain (общий код)
expect fun getPlatformName(): String

// androidMain
actual fun getPlatformName(): String = "Android"

// iosMain
actual fun getPlatformName(): String = "iOS"
```

Аналогия: в офисе висит табличка «Нужен переводчик» (expect). В офисе в Москве работает русскоговорящий (actual для России), а в Токио — японскоговорящий (actual для Японии).

### Что такое Material Design?

Material Design — это **дизайн-система от Google**: набор правил, как должны выглядеть и вести себя элементы интерфейса. Кнопки, карточки, переключатели, цвета, тени, анимации — всё это стандартизировано.

Аналогия: если UI — это дом, то Material Design — это стандарты строительства (ГОСТ). Они гарантируют, что кнопки будут одинакового размера, отступы — согласованы, а жесты — привычны пользователю.

**Material Design 3 (M3)** — третья версия системы. Главное нововведение — **Dynamic Color**: приложение автоматически подстраивает цвета под обои пользователя на Android 12+. Также добавлены новые компоненты и улучшена адаптивность (приложение красиво выглядит и на телефоне, и на планшете, и на складном устройстве).

## Сетевые термины (для мессенджеров и сетевых приложений)

Если ты строишь приложение, которое общается по сети (мессенджер, чат, P2P-клиент), тебе обязательно встретятся эти термины. Объясняю их простым языком.

### Что такое протокол?

Протокол — это набор правил, по которым две программы общаются друг с другом. Аналогия: протокол — как дипломатический этикет. Когда два посла встречаются, они знают, кто первый здоровается, как передать письмо, как ответить. Так же и программы: протокол определяет, в каком формате отправлять данные, как подтверждать получение, что делать при ошибке.

### HTTP / HTTPS

**HTTP** (HyperText Transfer Protocol) — самый распространённый протокол в интернете. Когда ты открываешь сайт — браузер использует HTTP, чтобы попросить сервер прислать страницу. **HTTPS** — то же самое, но с шифрованием (буква S = Secure).

HTTP работает по модели «запрос → ответ»: клиент спрашивает, сервер отвечает. Это неудобно для мессенджеров, потому что сервер не может сам «протолкнуть» новое сообщение клиенту — клиент должен сам спрашивать.

### WebSocket

**WebSocket** — протокол, который создаёт **постоянное двустороннее соединение** между клиентом и сервером. В отличие от HTTP (спросил → ответил → разорвал связь), WebSocket держит канал открытым. Любая сторона может отправить сообщение в любой момент.

Аналогия: HTTP — как почта: написал письмо, отправил, ждёшь ответ. WebSocket — как телефонный звонок: оба на линии, и каждый может говорить когда хочет.

Для мессенджера WebSocket идеален: сервер может мгновенно прислать новое сообщение, как только оно появилось.

### QUIC

**QUIC** — современный протокол от Google (теперь стандарт IETF), работает поверх UDP. Он был создан, чтобы решить проблемы TCP: медленное установление соединения и «head-of-line blocking» (когда потеря одного пакета блокирует все остальные).

Аналогия: если TCP — это поезд, где все вагоны ждут друг друга, то QUIC — это мотоциклисты: каждый едет независимо, и если один отстал, остальные продолжают путь.

Преимущества QUIC:
- **Быстрое соединение:** 1 round-trip вместо 3 (как в TCP + TLS)
- **Мультиплексирование без блокировок:** несколько потоков данных, потеря пакета в одном не тормозит другие
- **Встроенное шифрование:** TLS прямо внутри протокола
- **Миграция соединения:** если сменил сеть (Wi-Fi → мобильный интернет), соединение не рвётся

Идеален для мессенджеров, где важна скорость и стабильность при плохом интернете.

### P2P (Peer-to-Peer)

**P2P** — прямое соединение между двумя устройствами, без промежуточного сервера. Аналогия: обычный телефонный звонок — вы говорите напрямую, без оператора на линии.

В отличие от клиент-серверной модели (все идут через центральный сервер), в P2P устройства общаются напрямую. Плюсы: приватность (сервер не видит данные) и скорость (нет промежуточного звена). Минусы: сложно подключиться, если устройства за NAT.

### NAT (Network Address Translation)

**NAT** — технология, которая позволяет множеству устройств в домашней/офисной сети выходить в интернет через один общий IP-адрес. Твой роутер делает NAT: у него один внешний IP, а у твоего телефона, ноутбука и умной лампочки — внутренние (192.168.x.x).

Проблема NAT для P2P: устройство за NAT не может быть «достучано» снаружи — роутер не знает, какому устройству передать входящее соединение. Это как жить в многоквартирном доме: курьер знает адрес дома, но не номер квартиры.

### STUN / TURN

**STUN** и **TURN** — протоколы для обхода NAT при установлении P2P-соединения.

**STUN** (Session Traversal Utilities for NAT) — помогает устройству узнать свой внешний IP-адрес (какой его видят в интернете). Аналогия: STUN — как услуга «узнай свой номер телефона». Ты звонишь на специальный сервер, он говорит: «Ты звонишь с номера 203.0.113.5:49200». Теперь ты можешь передать этот номер собеседнику.

**TURN** (Traversal Using Relays around NAT) — если прямое P2P-соединение невозможно (NAT слишком строгий), данные идут через промежуточный сервер-ретранслятор. Аналогия: если два человека не могут встретиться лично (NAT мешает), они общаются через курьера (TURN-сервер). Курьер видит данные, но зато связь работает.

### TLS (Transport Layer Security)

**TLS** — протокол шифрования данных при передаче по сети. Когда ты видишь «https://» в браузере — это значит, что данные зашифрованы с помощью TLS.

Аналогия: TLS — как запечатанный конверт. Без TLS — как открытка: любой почтальон может прочитать. С TLS — только получатель может вскрыть и прочитать.

TLS обеспечивает:
- **Шифрование:** данные не может прочитать перехватчик
- **Аутентификацию:** ты точно общаешься с тем сервером, с каким думаешь
- **Целостность:** данные не были изменены по пути

### Шифрование: симметричное и асимметричное

**Симметричное шифрование** — один и тот же ключ используется для шифрования и расшифровки. Аналогия: замок с одним ключом. Ты запираешь дверь — тот же ключ открывает. Быстро, но как безопасно передать ключ собеседнику?

**Асимметричное шифрование** — два ключа: публичный (для шифрования) и приватный (для расшифровки). Аналогия: почтовый ящик. Любой может опустить письмо (зная публичный ключ — адрес ящика), но достать может только владелец (имея приватный ключ — ключ от ящика).

На практике используется комбинация: асимметричное шифрование — чтобы безопасно обменяться ключами, а потом симметричное — для быстрой передачи данных.

### Сквозное шифрование (End-to-End Encryption, E2EE)

**E2EE** — данные зашифрованы на устройстве отправителя и расшифрованы только на устройстве получателя. Даже сервер-посредник видит только «мусор», а не содержимое.

Аналогия: ты кладёшь письмо в сейф, отправляешь сейф курьером. Курьер не может открыть сейф — ключ только у получателя. Даже если курьер (сервер) будет перехвачен, содержимое останется секретом.

В ReMatch Chat: Direct-talk использует E2EE — Администрация не имеет доступа к содержимому, потому что ключи шифрования есть только у участников разговора.

### Сжатие данных (lz4)

**lz4** — алгоритм быстрого сжатия данных. Он не сжимает максимально плотно, но делает это **очень быстро** — идеально для синхронизации в реальном времени, где важна скорость, а не максимальная экономия байтов.

Аналогия: lz4 — как быстрое складывание одежды в чемодан. Не идеально компактно (как вакуумный пакет), зато в 10 раз быстрее.

### mDNS (Multicast DNS)

**mDNS** — протокол обнаружения устройств в локальной сети без центрального DNS-сервера. Устройства сами «кричат» в сеть: «Я — телефон Алисы, мой адрес 192.168.1.42!», и другие устройства могут их найти.

Аналогия: mDNS — как перекличка в классе. Учитель не нужен — каждый сам называет своё имя, и все слышат.

Используется в ReMatch Chat для синхронизации устройств в одной Wi-Fi-сети без intermediation сервера.

| **instance** | Конкретный объект, созданный по чертежу (классу) |
| **interface** | Контракт: класс, реализующий интерфейс, обязан реализовать его методы |
| **JVM** | Виртуальная машина Java — запускает скомпилированный Kotlin/Java |
