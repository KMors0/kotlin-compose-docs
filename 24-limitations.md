# Ограничения и контраргументы

> Объективный анализ ограничений стека Kotlin + Compose Multiplatform + M3 — с опорой на исследования и отзывы разработчиков.

---

## 24.1. Производительность на iOS

Несмотря на стабильный статус Compose Multiplatform для iOS, вопросы производительности остаются предметом обсуждения. По результатам исследования KTH Royal Institute of Technology, производительность Compose Multiplatform на iOS сопоставима с нативным SwiftUI для типичных сценариев (списки до 100 элементов, стандартные переходы), однако при экстремальных нагрузках (списки 1000+ элементов, частые обновления) наблюдается увеличенное использование памяти и время кадра по сравнению с нативным решением.

Разработчик Elijah Semyonov из команды JetBrains в своём докладе на KotlinConf 2024 продемонстрировал, что цель Compose Multiplatform для iOS — «абсолютно плавная производительность даже на дисплеях с высокой частотой обновления», и что команда активно работает над оптимизацией рендеринга через Metal backend.

**Практическая рекомендация:** Для приложений с высоконагруженными UI (финансовые тикеры, стриминг данных, сложные анимации) рекомендуется провести prototyping и нагрузочное тестирование на iOS перед полным внедрением.

## 24.2. Зрелость веб-платформы

Compose Multiplatform для веба, хотя и достиг бета-статуса в версии 1.9.0, всё ещё имеет ограничения:

- **SEO:** Compose для веба рендерит UI через Canvas/WebGL, что затрудняет индексацию контента поисковыми системами
- **Производительность:** начальная загрузка (initial load) может быть медленной из-за необходимости загрузить WebAssembly runtime и Skia
- **Экосистема:** меньшее количество готовых библиотек по сравнению с React/Vue/Angular
- **Интеграция:** хотя `WebElementView()` позволяет встраивать HTML-элементы, полноценная интеграция с существующими веб-фреймворками требует усилий

Согласно отзывам разработчиков, базовые функции работают хорошо, но для сложных веб-приложений (e-commerce, контентные платформы) Compose для веба пока не является оптимальным выбором.

## 24.3. Экосистема и сторонние библиотеки

Хотя экосистема Compose Multiplatform быстро растёт, она всё ещё уступает по зрелости React Native и Flutter в ряде областей. Картографические SDK (Google Maps, MapKit), SDK оплаты (Stripe, Square), аналитические библиотеки и специализированные компоненты часто требуют написания платформоспецифичных обёрток или использования `expect`/`actual` паттернов.

С другой стороны, базовые задачи (сетевые запросы через Ktor, сериализация через kotlinx.serialization, локальные базы данных через SQLDelight) имеют первоклассную поддержку в KMP-экосистеме.

**Оценка зрелости по категориям:**

| Категория | Зрелость | Примечание |
|-----------|----------|------------|
| Сетевые запросы (Ktor) | ✅ Отлично | Полная KMP-поддержка |
| Сериализация (kotlinx.serialization) | ✅ Отлично | Без рефлексии, работает везде |
| Базы данных (SQLDelight) | ✅ Отлично | Типобезопасные SQL-запросы |
| DI (Koin) | ✅ Хорошо | Работает на всех платформах |
| Карты | ⚠️ Ограниченно | Требуются платформенные обёртки |
| Платежи (Stripe) | ⚠️ Ограниченно | Только через expect/actual |
| Аналитика | ⚠️ Ограниченно | Часто требует платформенных SDK |

---

## 24.4. Адаптация команды и кривая обучения

Переход на Compose Multiplatform требует от команды освоения нескольких новых технологий одновременно: Kotlin (если команда пришла с Swift/Objective-C), декларативная парадигма Compose (если команда привыкла к UIKit/View system), и концепции KMP. Однако для Android-разработчиков порог входа минимален, а для iOS-разработчиков — сопоставим с изучением SwiftUI.

Согласно survey от JetBrains, основной мотивацией для внедрения KMP является **снижение дублирования кода** и **повышение согласованности** бизнес-логики между платформами, а не полная замена нативных разработчиков кроссплатформенными.
