# Руководство по участию в развитии документации

Спасибо за интерес к улучшению этой документации! Ниже описаны правила и процесс.

---

## Как сообщить об ошибке

1. Откройте Issue в репозитории с меткой `bug`.
2. Укажите: главу, строку (если возможно), ожидаемое поведение и фактическое.
3. Если есть техническая ошибка в коде — приложите минимальный reproducible example.

## Как предложить улучшение

1. Откройте Issue с меткой `enhancement`.
2. Опишите, что хотите добавить/изменить и почему.
3. Если возможно — набросайте структуру будущей главы.

## Как добавить главу

### Стиль

- Язык: русский.
- Формат: Markdown с GitHub Flavored syntax.
- Шаблон главы: заголовок с `>` эпиграфом, `---` разделители между секциями, нумерация вида `XX.Y`.
- Код: Kotlin, с import-выражениями, реальные компилируемые примеры.
- После каждого блока кода — 2–4 предложения объяснения.
- В конце главы — навигационные ссылки: `⬅️ [← ...] | [К содержанию](README.md) | [... →] ➡️`

### Нумерация

- Новые практические главы: продолжают нумерацию.
- Глубокие погружения: 20–27.
- Справочные материалы: в подкаталогах `metanit/`, `kotlinlang/`.

### Версионирование примеров

В шапке сложных глав указывайте версии, на которых проверен код:

```
> Проверено на Kotlin 2.1.0, Compose Multiplatform 1.7.3, Material 3.
```

### Обновление README.md

При добавлении новой главы обязательно обновите `README.md`: добавьте строку в соответствующую таблицу содержания.

---

## Форматирование кода

- Используйте `kotlin` для блоков кода (не `java`, не `groovy`).
- Gradle-файлы помечайте как `kotlin`.
- YAML помечайте как `yaml`.
- Максимальная ширина строки кода: 100 символов.
